# CS-230-13421-M01-Operating-Platforms

1. Briefly summarize The Gaming Room client and their software requirements. Who was the client? What type of software did they want you to design?

The client was The Gaming Room. They wanted a web-based version of their game, Draw It or Lose It, that could work across multiple platforms instead of only Android. The software needed to support multiple teams and players, use unique game and team names, prevent duplicate images and keep the game organized through one main game service.

2. What did you do particularly well in developing this documentation?

I think I did well explaining the structure of the system in a simple way. I broke down how the classes connect, why the singleton pattern was useful, and how the design helps prevent duplicate games, teams, players, and names.

3. What about the process of working through a design document did you find helpful when developing the code?

The design document helped because it made me think through the system before coding it. It gave me a plan for what classes were needed, how they should connect, and what requirements had to be included. It made the code easier to organize instead of just building without direction.

4. If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?

I would revise the evaluation section. I would improve it by making the comparisons between Mac, Linux, Windows, and mobile devices more detailed and easier to understand. I'd also explain more clearly why Linux is the best server for this type of game.

5. How did you interpret the user’s needs and implement them into your software design? Why is it so important to consider the user’s needs when designing?

I interpreted user's needs by focusing on what the game had to do: support multiple players, multiple teams, unique names, and access from different devices. I implemented that by using client/server design, singleton pattern, inheritance, and encapsulation. Considering user needs is important because the software has to actually solve the client’s problem, not just function.

6. How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?

I approached the design by looking at the requirements first, then deciding what structure would best support them. I used object-oriented programming, UML, inheritance, encapsulation, and the singleton pattern to keep the system organized. In the future, I'd use the same strategy: review client’s needs, identify the main objects, create a plan, then build the code around that plan.
